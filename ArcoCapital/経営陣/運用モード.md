## 1. 3モードの位置づけ

| モード | 略称 | 承認ゲート | 外部送信 | 有料クレジット | 用途 |
|:-----:|------|:----------:|:-------:|:-------------:|------|
| **A** | Dry-run    | 不要 | ❌ | ❌ | 設計検証・障害時フォールバック |
| **B** | Semi-live  | **必須** | ✅ | ✅ | 通常運用・Phase 0 標準 |
| **C** | Full-live  | スキップ | ✅ | ✅ | ゲート通過部署のみ・Phase 2〜 |

### A: Dry-run (フェイルセーフ)
- 草案の生成のみ。X投稿・note公開・Renoise有料消費を全面禁止
- 障害発生時の Kill-switch 発火先
- 新機能 PoC の初期ステージ

### B: Semi-live (現在の標準) ← **Phase 0 で全事業部がここ**
- 草案生成 → オーナー承認 → 送信の2段階
- 承認キュー滞留上限: **15分**
- DM返信 SLO: **24時間以内**
- 誤投稿検知 SLO: **30分以内**

### C: Full-live (将来目標)
- 承認ゲートをスキップし全自動実行
- **事業部単位で付与**。全社一括ではない
- 付与条件: **直近4週のKPI達成 + 炎上0件** (COO 拘束条件)

## 2. モード昇格・降格フロー

```
   ┌──────────┐  Phase 1 ゲート4条件 クリア   ┌──────────┐
   │  Mode B  │ ───────────────────────────→ │  Mode C  │
   │  (既定)  │                               │ (事業部別)│
   └──────────┘                               └──────────┘
        ▲                                          │
        │  炎上累計2件 / 誤投稿1件                 │
        └──────────────────────────────────────────┘
                                                   │
                     ┌────────┐ Kill-switch ←──────┘
                     │ Mode A │ 発火
                     └────────┘
```

### 昇格トリガー (B → C)
Phase 1 ゲート判定会議 (2026-04-25) で**事業部単位**に以下4条件を全クリア:

1. ✅ `cinematic-everyday-01` 完成 (SNS動画事業部の場合)
2. ✅ DM返信SLO 24h の Week1 達成率 100%
3. ✅ 監視ダッシュボード稼働 (エラー率 / SLO / コスト可視化)
4. ✅ Week1 で炎上・誤投稿 0件

### 降格トリガー (C → B / B → A)
- 炎上累計 **2件** で全事業部 Mode B 恒久化 (CFO 拘束条件)
- 誤投稿 **1件** で該当事業部 Mode A 差し戻し
- 予算ハードキャップ超過 (API ¥1,500/日, Renoise ¥10,000/週) で Kill-switch 自動発火
- オーナー手動発火 (`python -m operations.kill_switch --trigger manual`)
